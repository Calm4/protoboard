# Protoboard — руководство для нового чата (передача состояния)

Этот файл — краткая, но полная карта проекта: что это, как устроено, что готово,
что в работе и на что обратить внимание. Прочитай его первым в новом чате.

> Пользователь — не сильный технически. Объяснять простыми словами, идти по шагам,
> ключи/SQL/деплой он делает сам (агент подсказывает, где и что). Любит работать по
> «брифу» и поэтапно.

---

## 1. Что это

**Protoboard** — веб-трекер задач и багов для команды (~15–20 чел.) по прототипам
мобильных игр. Упрощённая Asana: один проект = один прототип. Доступ по ссылке,
изменения видны всем в реальном времени.

## 2. Стек и инфраструктура

- **Фронтенд:** React + Vite (одностраничное приложение). Сборка `npm run build`.
- **Бэкенд:** Supabase (Postgres + Realtime + Storage). Project ref: `thlcioruvtiucwtzqure`.
- **Хостинг:** **GitHub Pages** → https://calm4.github.io/protoboard/
  - Репозиторий: `github.com/Calm4/protoboard` (личный аккаунт пользователя).
  - Авто-деплой: GitHub Actions (`.github/workflows/deploy.yml`) на каждый push в `main`.
  - **НЕ Cloudflare Pages** — `*.pages.dev` заблокирован у пользователя в регионе.
- **Ключи Supabase:** publishable/anon (публичный) лежит в `.env.local` (локально) и
  закоммичен в `.env.production` (для сборки в CI). Это нормально — ключ публичный,
  доступ ограничивают политики RLS. Секретный ключ НИГДЕ не используется.
- **Деплой делает пользователь:** через **GitHub Desktop → Push**. Агент пушить НЕ может
  (в окружении агента нет авторизации к GitHub). Коммитить агент может; пушит — пользователь.

## 3. Структура кода

```
src/
  App.jsx                 — главный компонент: состояние UI, открытие задачи по ссылке,
                            экраны загрузки/ошибки, сборка экранов
  hooks/
    useProjects.js        — ВСЯ работа с данными: загрузка (с повтором+таймаутом),
                            Realtime, CRUD проектов/задач/статусов, порядок, скриншоты
    useAuth.js            — задел под вход (AUTH_REQUIRED=false, вход выключен)
  lib/
    supabase.js           — клиент Supabase (читает .env)
    image.js              — сжатие картинок + скачивание
    color.js              — hexToRgba (тинты для плашек)
  components/
    ProjectGrid.jsx       — главная: список проектов + архив
    ProjectView.jsx       — экран проекта: шапка, поиск, фильтр платформы, доска/список
    Board.jsx             — доска: колонки=статусы, dnd карточек и колонок, верх. скролбар
    TaskList.jsx          — список: группы по статусам, инлайн-выпадашки, dnd
    TaskPanel.jsx         — панель задачи: поля, скриншоты, #N + «скопировать ссылку»
    StatusMenu.jsx        — меню статуса (переименовать/цвет/удалить), общий для доски/списка
    Editable.jsx          — EditableInput/EditableTextarea (правка по Enter/blur, авто-рост)
    ColorSwatches.jsx     — палитра цветов (проект/статус)
    NewProjectModal.jsx   — окно создания проекта (имя + цвет)
    LoginScreen.jsx       — экран входа (показывается только если AUTH_REQUIRED=true)
  constants.js            — справочники: PRIORITIES, PLATFORMS, DEFAULT_STATUSES,
                            PROJECT_COLORS, PRIO_UI/PLAT_UI (цвета плашек)
  styles.js               — все стили (строка CSS, классы .pb-*)
supabase/                 — SQL-миграции (см. раздел 6)
docs/                     — этот файл + оригинальный прототип/бриф
```

**Ключевая идея архитектуры:** все данные идут через `useProjects.js`. Компоненты
получают данные и колбэки оттуда. Поля задачи в приложении: `title` (главный текст,
бывш. название+описание), `notes`, `priority` (high/med/low), `status` (id из статусов
проекта), `platform` (ios/android/both), `version`, `order` (sort_order, дробный),
`num` (#N), `shots[]`. Проект: `name`, `build` (версия), `archived`, `color`, `statuses[]`
(массив {id,label,color}), `tasks[]`.

## 4. Что готово (всё работает)

- Проекты: создание (с выбором цвета), архив/возврат, переименование, цвет (полоска/кружок),
  редактируемая версия проекта.
- Задачи: доска (drag между колонками **и** перестановка внутри колонки) и список
  (группировка по статусам, показ всех статусов включая пустые, dnd, инлайн-смена
  статуса/приоритета/платформы выпадашками).
- **Свои статусы у каждого проекта**: добавить (+), переименовать, цвет, перетащить
  порядок, удалить (задачи уезжают в первую колонку). База — три статуса по умолчанию.
- Объединённое поле названия (многострочное, авто-растёт под текст) + «Доп. условия».
- Скриншоты: загрузка в Storage с **сжатием** (≤1920px, JPEG), просмотр на весь экран,
  кнопка «Скачать». При удалении задачи файлы тоже чистятся.
- **Realtime**: изменения видны всем сразу.
- Поиск задач (по названию и версии) + фильтр по платформе (Все/iOS/Android).
- Плашка версии (обводка). Версии у части задач извлечены из текста.
- **#N (id для багов)** + ссылка `#task=<uuid>` («Скопировать ссылку» в панели) —
  открывает конкретную задачу у любого, кто перейдёт. Номер = max+1 в проекте,
  переиспользуется при удалении верхнего.
- Надёжная загрузка: повтор до 3 раз с таймаутом 8с на попытку; экраны «Загрузка…» и
  «Не удалось связаться с базой… [Повторить]» вместо пустого/вечно-грузящегося экрана.
- **Ctrl+Z / ⌘+Z (отмена действий)**: откатывает последнее действие — правку полей задачи,
  перемещение/перестановку, создание/удаление задачи, операции со статусами (добавить/
  переименовать/цвет/порядок/удалить), имя/цвет/версию/архив проекта, создание проекта.
  Внизу появляется уведомление «Отменено: …». История — на текущей вкладке (у каждого
  своя, чужие правки не трогает), глубина до 40 шагов. В текстовых полях клавиша не
  перехватывается — там работает обычная отмена ввода браузера. Реализация: «стопка»
  обратных операций в useProjects.js (`undo`), горячая клавиша и тост — в App.jsx.
- Задел под вход (useAuth, AUTH_REQUIRED=false) — включается без переделки (см. ниже).

## 6. SQL-миграции (важно!) — что выполнено, что нет

Файлы в `supabase/`. Выполняются в Supabase → SQL Editor → Run. Порядок: SQL ПЕРЕД push
кода, который от него зависит.

| Файл | Что делает | Статус |
|---|---|---|
| `schema.sql` | таблицы projects/tasks/attachments + RLS + realtime | ✅ выполнен |
| `storage.sql` | бакет `screenshots` + политики | ✅ выполнен |
| `round2.sql` | колонка `color` у проектов + перенос описаний в названия | ✅ выполнен |
| `statuses.sql` | колонка `statuses` + снятие CHECK на tasks.status | ✅ выполнен |
| `order.sql` | `sort_order` → double precision (для перестановки) | ⚠️ проверить: если перестановка на сайте работает — выполнен |
| `task-num.sql` | колонка `num` + проставить номера | ❌ **НЕ выполнен — выполнить ДО push последнего коммита** |
| `auth-later.sql` | ужесточение RLS при включении входа | ⏸️ только когда включат вход |

## 9. На что обратить внимание / нюансы

- **Сеть к Supabase у пользователя бывает капризной** (Supabase за Cloudflare, регион).
  Лечится повтором+таймаутом (сделано). Если станет совсем плохо у команды без VPN —
  крайнее решение: свой домен + проксирование Supabase через него.
- **У окружения агента та же капризная связь с Supabase** — REST/preview-проверки могут
  падать с таймаутом. Это не баг сайта. Для прямых запросов к Supabase из Python
  использовать `ssl._create_unverified_context()` и несколько повторов; ключ — publishable
  из `.env.local`; запускать с `dangerouslyDisableSandbox: true` при сетевых обрывах.
- **Включить вход** (когда понадобится): `AUTH_REQUIRED=true` в `src/hooks/useAuth.js` +
  в Supabase включить Email/Magic Link + выполнить `supabase/auth-later.sql`. Экран входа
  уже готов.
- **Передеплой/обновление сайта** = push в `main` (GitHub Desktop) → Actions соберёт сам.
- `num` считается на клиенте (max+1) — при одновременном создании двумя людьми возможна
  коллизия номера (редко, для маленькой команды ок). Ссылки на uuid от этого не страдают.
- **Ctrl+Z и скриншоты при удалении задачи.** Чтобы удаление задачи можно было отменить
  вместе с её скриншотами, файлы из Storage удаляются НЕ сразу, а отложенно — когда запись
  об удалении уходит со «дна» стопки отмены (после ~40 действий). Если удалить задачу со
  скриншотами и перезагрузить вкладку, не дойдя до очистки, файлы могут остаться мусором в
  Storage (редкий случай; чистится вручную). Удаление скриншота по отдельной кнопке в панели
  отменять НЕЛЬЗЯ — файл удаляется сразу.
- **Отмена — на текущей вкладке.** Стопка отмены живёт в браузере и очищается при
  перезагрузке; отменяются только свои действия этой сессии, не чужие правки из Realtime.
- Скриншоты — главный потребитель Storage (лимит 1 ГБ free). Сжатие включено; следить в
  Supabase → Settings → Usage.
