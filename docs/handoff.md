# Protoboard — руководство для нового чата (передача состояния)

Этот файл — краткая, но полная карта проекта: что это, как устроено, что готово,
что в работе и на что обратить внимание. Прочитай его первым в новом чате.

> Пользователь — не сильный технически. Объяснять простыми словами, идти по шагам,
> ключи/SQL/деплой он делает сам (агент подсказывает, где и что). Любит работать по
> «брифу» и поэтапно.

---

## 1. Что это

**Protoboard** — веб-трекер задач и багов для команды (~15–20 чел.) по прототипам
мобильных игр. Упрощённая Asana: один проект = один прототип. Доступ по ссылке,
изменения видны всем в реальном времени.

## 2. Стек и инфраструктура

- **Фронтенд:** React + Vite (одностраничное приложение). Сборка `npm run build`.
- **База данных:** **Firebase Firestore** (заменил Supabase в июне 2026).
  - Проект: `protoboard-fb` (firestore region: `eur3`).
  - Ключи — в `.env.local` (локально) и `.env.production` (для CI/CD, закоммичено).
    apiKey Firebase публичный по задумке — защита через Firestore Security Rules.
  - **Rules:** файл `firestore.rules` в корне репо — источник правды, но применяется
    вручную через Firebase Console (Firestore Database → Rules → вставить содержимое
    файла → Publish), т.к. Firebase CLI на машине агента не установлен/не авторизован.
    **Если правишь `firestore.rules` в коде — не забудь скопировать текст в консоль,
    иначе изменения не подействуют.** Модель (уточнена 2026-07-01): читать/писать может
    только вошедший пользователь; создавать/архивировать проекты, менять их поля и
    управлять статусами может любой вошедший — `admin` отличается только правом
    **удалить** проект (необратимо) и назначать роли другим. Менять себе роль нельзя.
  - Коллекции: `projects`, `tasks`, `attachments`, `activity`, **`users`** (профиль +
    роль `admin`/`user`, создаётся автоматически при первом входе через Google, по
    умолчанию `"user"` — первого admin нужно проставить вручную в консоли Firebase;
    с 2026-07-01 также `customName`/`position` — см. раздел 4).
- **Хостинг:** **GitHub Pages** → https://calm4.github.io/protoboard/
  - Репозиторий: `github.com/Calm4/protoboard` (личный аккаунт пользователя).
  - Авто-деплой: GitHub Actions (`.github/workflows/deploy.yml`) на каждый push в `main`.
  - **НЕ Cloudflare Pages** — `*.pages.dev` заблокирован у пользователя в регионе.
- **Скриншоты:** хранятся как **base64 data URL прямо в Firestore** (коллекция `attachments`).
  Firebase Storage не используется (требует платный Blaze-план). Cloudinary также заблокирован
  в регионе пользователя. Лимит Firestore на документ — 1 МБ; сжатие до ≤1920px JPEG
  держит большинство скриншотов в пределах 800 КБ base64.
- **Деплой делает пользователь:** через **GitHub Desktop → Push**. Агент пушить НЕ может.
  Коммитить агент может; пушит — пользователь.

## 3. Структура кода

```
src/
  App.jsx                 — главный компонент: состояние UI, открытие задачи по ссылке,
                            экраны загрузки/ошибки, сборка экранов
  hooks/
    useProjects.js        — ВСЯ работа с данными: Firestore onSnapshot, CRUD
                            проектов/задач/статусов/тегов/участников, скриншоты (lazy
                            load + upload), кэш скриншотов (shotsCacheRef), отмена действий
    useAuth.js            — вход через Google (Firebase Auth), читает/создаёт роль +
                            customName/position в Firestore users/{uid}
    useUsers.js            — живой каталог users (для резолва имён/поиска людей)
  lib/
    firebase.js           — клиент Firebase (читает .env VITE_FIREBASE_*)
    image.js              — сжатие картинок (canvas → JPEG) + скачивание
    color.js              — hexToRgba (тинты для плашек)
    people.js              — personName(u)/personPosition(u) — резолв customName/
                            displayName/email и должности из каталога users
  components/
    ProjectGrid.jsx       — главная: «Мои проекты» / «Все проекты» (можно присоединиться),
                            архив, глобальный поиск (задачи + проекты)
    ProjectView.jsx       — экран проекта: шапка, инлайн-строка фильтров, доска/список
    ProjectSettingsModal.jsx — настройки проекта (⚙): название, цвет, градиент,
                            статусы (переименовать/цвет/порядок/удалить), теги, участники
    MembersModal.jsx        — участники проекта: список с удалением + поиск/добавление
    AssigneePicker.jsx      — поле «Исполнитель»: поиск и выбор из участников проекта
    ProfilePage.jsx          — полноэкранная страница профиля (не поп-ап): своё имя/
                            должность, «Мои проекты», «Мои задачи», выход
    OnboardingModal.jsx     — имя + должность при самом первом входе (можно пропустить)
    UserMenu.jsx / HeaderControls.jsx — кнопка профиля и переключатель темы в шапке
    Board.jsx             — доска: колонки=статусы, dnd карточек и колонок, верх. скролбар
    TaskList.jsx          — список: группы по статусам, инлайн-выпадашки, dnd
    TaskPanel.jsx         — панель задачи: поля, скриншоты, #N + «скопировать ссылку»
    StatusMenu.jsx        — меню статуса на самой колонке (переименовать/цвет/удалить)
    Editable.jsx          — EditableInput/EditableTextarea (правка по Enter/blur, авто-рост)
    ColorSwatches.jsx     — палитра цветов (проект/статус)
    NewProjectModal.jsx   — окно создания проекта (имя + цвет)
    LoginScreen.jsx       — экран входа (Google), показывается пока пользователь не вошёл
    DeleteProjectModal.jsx — модал удаления проекта (вводить «DELETE»)
    StatsView.jsx          — статистика: карточки, диаграммы, спарклайн
  constants.js            — PRIORITIES, PLATFORMS, DEFAULT_STATUSES, PROJECT_COLORS,
                            GRADIENTS (12 пресетов), EMPTY_FILTERS (showClosed:false),
                            POSITIONS (5 должностей для онбординга/профиля),
                            GLOBAL_TAGS (5 стандартных тегов — стартовый набор, не «вшит»)
  styles.js               — все стили (строка CSS, классы .pb-*)
scripts/
  migrate-to-firebase.js  — одноразовый скрипт миграции Supabase→Firebase (уже выполнен)
docs/                     — этот файл
```

**Ключевая идея архитектуры:** все данные идут через `useProjects.js`. Компоненты
получают данные и колбэки оттуда.

**Поля задачи в приложении:** `title`, `notes`, `priority` (high/med/low), `status` (id из
статусов проекта), `platform` (ios/android/both), `version`, `order` (дробный, sortOrder в
Firestore), `num` (#N), `tags[]`, `shots[]` (lazy), `shotsLoaded` (bool),
`activity[]` (lazy), `activityLoaded` (bool), `dueDate` (YYYY-MM-DD),
`assignee` (**uid участника проекта** для новых назначений с 2026-07-01, см. ниже — либо
свободный текст для старых задач), `completedAt` (timestamp ms | null — ставится при
попадании в последний статус), `closed` (bool — «выполнено» вручную, независимо от статуса).

**Поля задачи в Firestore:** `projectId`, `title`, `description`, `notes`, `priority`, `status`,
`platform`, `version`, `sortOrder`, `num`, `tags[]`, `createdAt`, `dueDate`, `assignee`,
`completedAt`, `closed`.

**Проект:** `name`, `build`, `archived`, `color`, `statuses[]` (`{id,label,color}`),
`customTags[]`, **`hiddenTags[]`** (какие из `GLOBAL_TAGS` скрыты именно на этом
проекте — см. раздел 4), `gradient` (CSS gradient string | ""), **`members`** (`string[]`
uid участников — `undefined` у старых проектов до бэкфилла, `[]` легитимно «все вышли»).

**User (коллекция `users`, id = uid из Firebase Auth):** `displayName`, `email`, `photoURL`,
`role` (`"admin"` | `"user"`, по умолчанию `"user"`), `createdAt`, **`customName`** (своё
имя, необязательно, перекрывает `displayName` из Google — резолвится через `personName()`
в `lib/people.js`), **`position`** (одна из `POSITIONS` в `constants.js`, необязательно).

**Activity (коллекция `activity`):** `projectId`, `taskId`, `action` (string), `timestamp` (ms),
`authorName` (имя/почта вошедшего пользователя на момент действия, может быть `null` для
старых записей до 2026-07-01). Загружается лениво при открытии задачи.

**Attachment (коллекция `attachments`):** `taskId`, `name`, `data` (base64 data URL), `createdAt`.

## 4. Что готово (всё работает)

- **Вход через Google + роли (admin/user):** без входа приложение не открывается
  (`LoginScreen.jsx`). Роль хранится в `users/{uid}.role`. Обычный `user` может ВСЁ:
  создавать/архивировать/возвращать проекты, управлять статусами (добавить/переименовать/
  цвет/удалить/порядок колонок), работать с задачами (создание/редактирование/удаление,
  скриншоты, теги, чекбокс «выполнено», история). `admin` отличается только двумя вещами:
  может **удалить** проект (только из архива, необратимо, кнопка в `ProjectGrid.jsx`) и
  назначать роли другим пользователям (пока вручную через консоль). Кнопка удаления
  скрывается в интерфейсе по роли (`App.jsx` → `isAdmin`), но **источник правды —
  Firestore Rules** (`firestore.rules`), их нужно вручную опубликовать в консоли Firebase
  (см. раздел 2). Назначить кому-то роль `admin`: Firebase Console → Firestore Database →
  коллекция `users` → найти документ по uid (email виден в поле `email`) → поменять
  `role` на `"admin"`.
- **Единая шапка (главная + внутри проекта):** слева — логотип «Protoboard» (клик → на
  главную, одинаково на обоих экранах), справа — переключатель темы и кнопка профиля.
  Общий кусок — `HeaderControls.jsx`.
- **Профиль — отдельная страница (`ProfilePage.jsx`), не поп-ап** (с 2026-07-01, раньше
  был модал). Открывается из кнопки в шапке (`UserMenu.jsx` → `App.jsx` → `profileOpen`,
  третий экран наравне с главной/проектом, поддерживает кнопку «Назад» браузера).
  Аватар, роль (бейдж), **редактируемые имя и должность** (см. ниже), сворачиваемые
  секции «Мои проекты» и «Мои задачи» (по всем проектам сразу), кнопка «Выйти».
- **Своё имя и должность (`customName`/`position` на `users/{uid}`):** при самом первом
  входе через Google — `OnboardingModal.jsx` с полем имени и списком из 5 должностей
  (`POSITIONS` в `constants.js`: Programmer/Game Designer/Project Manager/QA Tester/
  Art Designer), можно пропустить — тогда останется имя из Google, должность не
  выставится. Менять оба поля можно в любой момент в `ProfilePage.jsx`. Везде, где
  показан человек (участники, исполнитель, карточка задачи — компактно, через `title`-
  тултип), имя резолвится через `personName()`/`personPosition()` (`lib/people.js`):
  `customName || displayName из Google || email`.
- **Участники проекта (`members[]` на проекте, «как в Slack», добавлено 2026-07-01):**
  все проекты публичные — любой вошедший видит их и может присоединиться (кнопка
  «Присоединиться» на карточке в разделе «Все проекты» на главной, или из глобального
  поиска — он теперь ищет и по названиям проектов, не только по задачам). Главная разбита
  на «Мои проекты» (где состоишь) и «Все проекты» (остальные, можно вступить). Кнопка
  «👥 Участники (N)» — теперь внутри настроек проекта (см. ниже) — открывает
  `MembersModal.jsx`: список участников с удалением (для себя — «Покинуть»), поиск и
  добавление новых из каталога пользователей (`useUsers.js`, живой список всех, кто
  когда-либо входил). Участники — организационная штука, не про безопасность (Firestore
  Rules НЕ ограничивают доступ по членству, специально, чтобы не усложнять и не
  блокировать никого). **Миграция старых 9 проектов** сделана без ручного скрипта: в
  `App.jsx` есть самомигрирующийся эффект — у первого залогиненного человека, открывшего
  обновлённое приложение, для каждого проекта без поля `members` подставляются uid всех,
  кто уже когда-либо входил (см. `useProjects.js` → `backfillMembers`).
- **Исполнитель — выпадающий список (`AssigneePicker.jsx`, добавлено 2026-07-01):** поле
  «Исполнитель» в задаче теперь список участников проекта с поиском, а не свободный текст.
  Хранит **uid**. Старые значения (текст, введённый раньше) не мигрированы — показываются
  как есть, если не резолвятся в участника.
- **Настройки проекта (`ProjectSettingsModal.jsx`, шестерёнка ⚙ — крайний правый элемент
  шапки проекта, добавлено 2026-07-01):** название (явная кнопка «Сохранить», не
  авто-коммит по blur — раньше можно было случайно переименовать проект), акцентный цвет
  и градиент фона, управление статусами (переименовать/цвет/переставить ▲▼/удалить/
  добавить — поверх того же `statusActions` API, что и на доске), управление **всеми**
  тегами проекта, кнопка входа в участников. Шапка проекта — три колонки: название слева,
  переключатель Статистика/Доска/Список по центру, действия (Выбрать/+Задача/⚙) справа.
- **Теги — полностью управляемые, включая стандартные (уточнено 2026-07-01):**
  `GLOBAL_TAGS` в `constants.js` — стартовый набор из 5 («Баг», «UI», «Краш», «Баланс»,
  «Перфоманс», раньше было 10), но НЕ «вшит намертво» — каждый проект может спрятать
  любой из них у себя через `project.hiddenTags[]` (не трогает сам глобальный список,
  только конкретный проект). `useProjects.js` → `removeProjectTag` сам разруливает:
  стандартный тег → в `hiddenTags`, свой (`customTags`) → удаляется насовсем; в обоих
  случаях тег снимается со всех задач проекта. Добавление нового тега без привязки к
  задаче — `addProjectTag`.
- **Инлайн-строка фильтров (добавлено 2026-07-01):** вместо кнопки «⚙ Фильтры», которая
  открывала попап, все фильтры (платформа/приоритет/статус/версия/№/дата) — сразу видимая
  строка над задачами. Теги — единственное, что осталось выпадающим списком (их может
  быть много). Приоритеты и платформы остаются глобальными константами (`PRIORITIES`/
  `PLATFORMS` в `constants.js`), НЕ per-project — сознательно отложено, см. раздел 7.
- **Вход в обход Google на localhost (только для теста, добавлено 2026-07-01):** на
  экране логина в dev-режиме (`npm run dev`) под кнопкой Google — «Войти без Google
  (только localhost)», анонимный вход Firebase (`useAuth.js` → `signInAnon`), настоящая
  сессия, не подделка. Гейт — `import.meta.env.DEV`, в `npm run build` кода кнопки
  физически нет в бандле (не просто скрыта). **Нужно один раз включить Anonymous в
  Firebase Console → Authentication → Sign-in method** — без этого кнопка есть, но вход
  падает с понятной ошибкой.
- Проекты: создание (с выбором цвета), архив/возврат, переименование, цвет, версия.
- **Удаление проекта:** только из архива. Подтверждение словом «DELETE».
- **Градиент-баннер проекта:** кнопка «🎨 Изменить фон» в шапке-баннере → пикер 12 пресетов.
  Градиент хранится в `projects.gradient`. Видно на карточке главного экрана и в баннере.
  Архивные карточки тоже показывают градиент (приглушённый через opacity).
- **Тёмная тема:** кнопка ☾/☀ в шапке главной. Класс `.pb.dark` переключает CSS-переменные.
  Состояние в `localStorage` (`pb-dark`).
- **Чекбокс «Выполнено» (Asana-style):** круглая кнопка на каждой задаче (доска и список).
  Отмечает задачу как закрытую (`closed:true`) независимо от статуса. По умолчанию при входе
  в проект фильтр `showClosed:false` скрывает такие задачи. Кнопка «✓ Выполненные (N)» в
  строке фильтров показывает их (зачёркнутые). В списке чекбокс в отдельной колонке с разделителем.
- **Дедлайн и исполнитель:** поля `dueDate` и `assignee` в TaskPanel. На карточках доски — бейджи
  «скоро» / «просрочено».
- **Bulk-действия:** кнопка «Выбрать», выделение нескольких задач, массовая смена статуса/
  приоритета, удаление. Плавающий BulkBar снизу.
- Задачи: доска (drag между колонками и внутри) и список (группировка, dnd, инлайн-редактирование).
- **Скриншоты:** lazy-загрузка (только при открытии панели), сжатие до ≤1920px JPEG,
  хранение base64 в Firestore. Спиннер во время загрузки, сообщение если файл не влез.
  Просмотр на весь экран, кнопка «Скачать». При удалении задачи файлы чистятся отложенно.
- **Realtime:** `onSnapshot` — изменения видны всем без перезагрузки.
- **Ctrl+Z / ⌘+Z (отмена):** откатывает создание/удаление/редактирование задач, статусов,
  проектов. История 40 шагов на вкладку. Тост «Отменено: …».
- **#N + ссылка на задачу:** «Скопировать ссылку» → `#task=<uuid>`. Переход по ссылке
  открывает задачу у любого.
- **Расширенная фильтрация:** платформа, приоритет, статус, версия, тег, номер #N, дата создания
  (пресеты + диапазон), showClosed. Счётчик активных фильтров.
- **Глобальный поиск:** строка на главной, ищет по всем проектам, открывает задачу напрямую.
- **Статистика:** вкладка открывается первой. Карточки (всего/открытые/на удержании/готово),
  разбивка по статусам/приоритету/платформе, двойной спарклайн «создано vs выполнено» за 30 дней.
  `isHoldStatus()` — ключевые слова для «заморозки» (hold/не фикс/не в билде и т.д.).
- **Теги:** добавление/удаление в TaskPanel, фильтр в строке фильтров — полная модель
  (стандартные + свои, все управляемые) описана в разделе про настройки проекта выше.
- **История изменений:** коллекция `activity`, анонимный лог, лениво при открытии задачи.
- **Мобильная оптимизация:** панель задачи на весь экран (≤768px).
- **PWA:** `public/manifest.json`, `theme-color`, `apple-mobile-web-app-capable`.

## 8. На что обратить внимание / нюансы

- **`closed` vs `completedAt`:** `closed` — ручная пометка «выполнено» (независимо от статуса).
  `completedAt` — автоматический timestamp при попадании задачи в последний статус. Оба поля
  независимы. Статистика использует `completedAt` для спарклайна.
- **`isHoldStatus(label)`** — дублируется в `StatsView.jsx` и `ProjectGrid.jsx`. Если нужно
  изменить ключевые слова заморозки, менять в обоих местах.
- **Пикер градиента** рендерится с `position:fixed` (вне карточки), т.к. `overflow:hidden` +
  `transform` на карточке обрезает `position:absolute` попапы.
- **Firebase вместо Supabase:** не путать. Supabase больше НЕ используется. Старые файлы
  `src/lib/supabase.js` и папка `supabase/` удалены 2026-07-01 (мёртвый код). Скрипт
  `scripts/migrate-to-firebase.js` (уже отработал) — единственное место в репо, где ещё
  остались ключи Supabase, оставлен как исторический артефакт.
- **Firestore лимит 1 МБ на документ:** большие скриншоты (>~900 КБ base64) не сохранятся.
  При ошибке пользователь увидит красный текст под блоком скриншотов. Решение — уменьшить
  качество сжатия в `lib/image.js` (параметр `quality`, сейчас 0.85) или `maxDim` (1920).
- **Кэш скриншотов (`shotsCacheRef`):** `Map<taskId, {shots, loaded}>` в `useRef`.
  Живёт только в текущей вкладке, сбрасывается при перезагрузке. Защита от гонки:
  uploading-превью кладётся в кэш сразу, чтобы `rebuild()` от Firestore его не перетёр.
- **Ctrl+Z и скриншоты при удалении задачи.** Удаление `attachments` из Firestore —
  отложенное (finalize при выходе записи со дна стека отмены). Если перезагрузить вкладку
  до финализации — записи в Firestore останутся, но после повторного удаления очистятся.
- **Отмена — на текущей вкладке.** Стопка живёт в браузере, очищается при перезагрузке.
- **Деплой:** агент коммитит, пушит пользователь через GitHub Desktop.
- **Роль admin-only в UI ≠ защита данных.** Единственное место, где сейчас используется
  `isAdmin` — кнопка «Удалить» в `ProjectGrid.jsx` (архив). Скрытие в интерфейсе — только
  удобство. Реальная защита — `firestore.rules`, и она действует только после публикации
  в консоли Firebase.
- **Первый admin** нужно проставить вручную (см. раздел 4) — без этого никто не сможет
  окончательно удалить проект (остальное всем и так открыто).
- **`project.members === undefined` vs `[]`** — не одно и то же. `undefined` = старый
  проект, ещё не мигрирован (эффект в `App.jsx` подставит всех известных пользователей
  при первом заходе кого угодно); `[]` = легитимно «участников нет». `ProjectGrid.jsx`
  временно считает `undefined` как «я участник», чтобы не мигало «0 проектов» до
  завершения бэкфилла — если трогаешь эту логику, не путать состояния.
- **Участники — не безопасность.** Никаких Firestore Rules под членство не заводили
  специально (см. раздел 4) — если понадобится реальная приватность проектов, это
  отдельная более рискованная задача (нужно менять Rules и, вероятно, схему загрузки
  задач в `useProjects.js`, которая сейчас грузит ВСЕ задачи одним `onSnapshot` без
  фильтра по проекту).
- **Вход без Google на localhost (добавлено 2026-07-01, Anonymous включён и проверен
  в этот же день):** на экране логина в dev-режиме (`npm run dev`) под кнопкой Google —
  «Войти без Google (только localhost)», анонимный вход Firebase (`useAuth.js` →
  `signInAnon`), настоящая сессия, не подделка. Гейт — `import.meta.env.DEV`, поэтому в
  `npm run build` кода кнопки физически нет в бандле (проверено — 0 вхождений строки
  после сборки), это не просто скрытая кнопка. Работает и для агента через
  `preview_*`-тулы — можно самому логиниться и смотреть живые данные проекта, не
  дожидаясь скриншотов от пользователя. У анонимных тестовых аккаунтов свой uid и
  пустой `users`-профиль каждый раз — не жалко, но они реально пишут в живой Firestore,
  будь аккуратен с созданием/удалением данных под такой сессией.
